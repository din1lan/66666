# 事務所工作台（原型）

React + Vite + Firebase 的單頁應用原型，涵蓋七個頁面：首頁總覽、案件與時效（含律師資料夾篩選 + 時效行事曆 + 開庭行事曆 + 末日提醒自動化）、結案案件表（取代原本的 Google Sheet）、客戶資料 CRM、線上卷宗（檔案上傳 + PDF 閱覽 + 檔名產生器）、財務與代墊款、以及 AI 閱卷與策略分析。側邊欄固定、右側內容依模組切換，資料共用同一個 Firestore 專案。

視覺設計採用深藍側邊欄 + 卡片式版面，AI 閱卷模組刻意做了來源分級（原文／推論／待核）與強制核對彈窗，讓幻覺風險在 UI 層就被攔一次，而不是全部丟給使用者自己判斷。

## 本機開發（使用 Firebase Emulator，免費、不需真實專案）

```bash
npm install
npm install -g firebase-tools   # 若尚未安裝
firebase emulators:start        # 開一個終端機視窗，保持執行
npm run dev                     # 另開一個終端機視窗
```

打開 `http://localhost:5173`。第一次啟動 emulator 會自動下載 Firestore/Storage/Auth 模擬器（需要網路），之後就會用本機快取。

> 這份原型的驗證是在一個網路白名單受限的沙盒環境裡完成的，該環境無法下載 Firebase Emulator 執行檔，所以 Firestore/Storage/Auth 的即時互動沒有在沙盒內跑過。已經驗證過的部分：`npm run build`（正式建置）、`npm run lint`（0 warning）、`npm test`（時效計算邏輯、案號簡寫/法院簡稱/工作日判斷/末日提醒、檔名產生規則、民國年日期轉換與結案資料正規化等單元測試，共 45 個測試全過，其中一項測試會直接核對 `closedCasesSeed.js` 匯入資料的筆數與律師別分佈，避免資料被意外改壞）。你在自己的電腦上執行 `firebase emulators:start` 時應該會有網路，屆時請照下面「上線前必須確認」的步驟，實際點過一次每個模組再放心使用，尤其是開庭行事曆、末日提醒、結案案件表匯入這幾個新功能——自動算出的日期和標題、匯入的歷史資料請務必人工核對過幾筆再信任。

## 接上真正的 Firebase 專案

```bash
cp .env.example .env.local
# 填入 Firebase Console → 專案設定 → 你的應用程式 SDK 設定
# 並把 VITE_USE_EMULATORS 設為 false
```

部署前記得同步規則與索引：

```bash

```

## 資料模型

- `clients` — 客戶基本資料（姓名、身分證字號/統編、聯絡方式）
- `cases` — 案件，透過 `clientId` 綁定客戶；`contractTotal` 記錄簽約總額
- `clientNotes` — 諮詢紀錄 / 備忘錄，透過 `clientId` 綁定
- `deadlines` — 時效紀錄，透過 `caseId` 綁定；`status` 為 `draft` 或 `confirmed`，見下方說明
- `billing` — 帳目（律師費 / 代墊款 / 其他支出），透過 `caseId` 綁定
- `files` — 檔案中繼資料（實際檔案存在 Storage 的 `cases/{caseId}/` 底下）；`suggestedFileName` 是行政人員用檔名產生器算出的建議檔名，僅供參考，不會自動改 Storage 裡的實際檔名
- `hearings` — 開庭通知（開庭行事曆），透過 `caseId`（選填）綁定；標題依「時間 律師-法院簡稱-當事人-案由-案號股別」規則自動組合
- `closedCases` — 結案紀錄（結案案件表），從 Google Sheet「結案簡表」匯入的歷史資料 + 往後的新登記共用同一個集合；`excludeFromReporting` 依備註關鍵字自動判斷是否計入事務所結案統計
- `_meta` — 目前只有 `_meta/closedCasesImport` 一筆，記錄歷史資料是否已經匯入過，避免同一份 Sheet 資料被重複匯入
- `aiConsent` — 每位員工（uid）對 AI 使用聲明的同意紀錄，取代原本畫面上「每次進入都彈窗、僅前端記憶」的做法

## 結案案件表（`/closed-cases`，取代 Google Sheet「結案簡表」）

行政人員原本用 Google Sheet 依律師分頁（陳/林/王/程律師）記錄每個結案案件的日期、當事人、案由、案號、結案文件、備註。這裡把該分頁的歷史資料整理成 `src/data/closedCasesSeed.js`（2026-07-08 從使用者提供的 Google Sheet 連結讀取，共 129 筆：陳律師 90、林律師 17、王律師 21、程律師 1），第一次進入頁面時可以按「匯入歷史資料」把這批資料寫進 Firestore（用 `_meta/closedCasesImport` 標記文件擋掉重複匯入）。**匯入之後這份 Google Sheet 就不再是資料來源** —— 之後新的結案案件請直接在「結案案件表」頁面用「新增結案登記」表單填寫，跟歷史資料共用同一套正規化規則（`src/lib/closedCaseImport.js`）：

- 案號欄位若含有多個案號（原始 Sheet 用換行或頓號分隔），會自動拆成陣列方便之後查詢。
- 備註含「不報結」或「不報事務所案件」等關鍵字的紀錄，會自動標記 `excludeFromReporting`，在頁面上以灰階顯示、也不計入結案統計，但資料本身仍然保留，不會被刪除或隱藏。
- 原始 Sheet 有「結案日期」與「結案日期(判決)」兩欄（互斥使用），匯入時優先採用「結案日期」，只有該欄位空白時才退回用判決日期，並保留 `closedDateType` 標記是哪一種——這是沿用 Sheet 兩欄並存設計時的推斷，不是行政人員明確交代的規則，如果事務所覺得優先順序應該反過來，改 `src/lib/closedCaseImport.js` 就好。
- 只匯入了 gid=1551802964 這個分頁（結案簡表）。Sheet 頁尾另外列出的「送達憑證」「筆錄調閱紀錄」「諮詢案件」等分頁，因為沒有取得對應連結，這次沒有涵蓋，需要的話請提供那幾個分頁的網址。

## 檔名產生器（線上卷宗頁，`/files`）

行政人員原本是把掃描檔丟給 Gemini 讀取內容（收文章、存底章、狀頭文字等）來產生檔名。目前先做了**表單式的決定論版本**（`src/data/fileNaming.js` + `FileNamingForm.jsx`）：行政人員自己看完掃描檔、判斷好日期/出狀收文/當事人/案由/狀別/案號股別等欄位後填入表單，系統照事務所的檔名規則自動組成正確格式的字串（含「親遞」「繕本」「開庭通知(MMDD-HHMM)」等規則），可以複製或存成該檔案的建議檔名——**完全不讀 PDF 內容，也不會自動重新命名 Storage 裡的實際檔案**，行政人員仍需自行在電腦上手動改檔名。

AI 自動讀取掃描檔內容（辨識存底章/收文章、擷取狀頭文字）是下一階段，架構骨架已經放在 `functions/`（`suggestFileNaming.js` + `README.md`），但**刻意保持未啟用**——需要事務所先確認 Anthropic API 的預算，並把 Firebase 從免費的 Spark 方案升級到 Blaze（依用量付費）方案，Cloud Functions 才能對外呼叫 AI API。`FileNamingForm.jsx` 裡的「AI 建議」按鈕目前是 disabled 狀態並附上原因說明，不是忘記做，是刻意留白等待事務所拍板。

## 末日提醒與開庭行事曆自動化（案件與時效頁）

這是把行政人員原本手動判斷、手動登記 Google Calendar 的流程搬進系統，涵蓋兩種行事曆：

- **開庭行事曆**（`hearings` collection + `HearingForm`/`HearingsCalendarView`）：填入律師、法院/機關全名、當事人、案由、案號股別、地點，系統自動組合行事曆標題與法院簡稱（`src/lib/courtAbbrev.js`）。未填時間 = 整日行程。
- **宥維工作（末日提醒）**（沿用既有 `deadlines` collection，新增 `ActionDeadlineForm`）：輸入收文日期與文件所載限期天數，系統自動計算「收文隔日起算第 N 日」的末日，若末日落在非工作日會自動順延至前一個工作日（`src/lib/taiwanHolidays.js`），並依限期天數分級表（`src/data/actionDeadline.js`）自動加開一筆提前提醒行程。

⚠️ **這兩個功能都是新的自動化邏輯，上線前請額外確認：**

1. **國定假日表目前只填了 2026 年（`taiwanHolidays.js` 的 `HOLIDAYS_STATUS = 'PARTIAL_2026_FILLED'`）。** 2026 年（民國115年）的 36 天國定假日已抄自行政院人事總處公告的「115年政府行政機關辦公日曆表」填入 `HOLIDAYS` 陣列，含元旦、農曆春節、228、兒童節/清明節、勞動節、端午節、中秋/教師節、國慶日、光復節、行憲紀念日。但**2026 以外的年度仍是空的**，農曆春節/端午/中秋等每年變動的假期、政府調整放假/補班日，都需要事務所每年年初自己填入當年度的 `HOLIDAYS` 陣列（可直接抄行政院人事總處公告）。在填入之前，遇到平日國定假日的末日不會自動順延，必須人工核對。
2. ~~提前提醒的天數分級表與原始範例互相矛盾。~~ **已於 2026-07-08 由事務所確認：採「分級表」版本（14~20日限期 → 提前5日提醒），原始範例文字「提前20日」認定為誤植，不採用。**（`src/data/actionDeadline.js` 的 `REMINDER_TIERS_STATUS = 'CONFIRMED_2026-07-08_BY_FIRM'`）表單中仍可手動覆寫提前天數，供個案調整使用。
3. **法院/機關簡稱是規則式猜測，不是每個機關都認得。** `abbreviateCourtName()` 只覆蓋「地方法院」「地方檢察署」「高等法院(分院)」三種型態；遇到勞動部、工程會等非法院機關會直接回傳原名，表單裡的簡稱欄位可手動修改，存檔前務必檢查。
4. **本次新增的 `hearings` collection 的 Firestore 規則與既有集合一樣，只檢查「有沒有登入」**，尚未做案件層級的存取控制，適用下方「上線前必須確認」第 2 項的同一個 gating checklist。

## AI 閱卷與策略分析模組（`/ai-review`）

左側是真的 react-pdf PDF 檢視器（讀取 Storage 裡的真實檔案），右側是分析面板。**右側的三個分析面板目前是靜態示範內容**，還沒有接上真正會讀取你上傳卷宗的 AI 模型——頁面上方有一條黃色提示條清楚說明這件事，避免有人誤以為那是系統真的分析了他們上傳的檔案。

引用徽章（「卷宗 p.X」）點下去只能跳轉到 PDF 的「頁」，沒辦法像原始 HTML 設計稿那樣精準跳到某一行並反白——react-pdf 是用 canvas 畫頁面，不是逐行的 HTML 元素，要做到逐行溯源需要額外做文字圖層比對，這是之後真正接上 AI 分析引擎時要一併解決的工程問題，不是隨手能補的小事。

「待核」（生成內容）徽章點下去會強制彈窗，連結到全國法規資料庫與司法院裁判書查詢系統。首次使用會要求勾選三項聲明，寫入 `aiConsent/{uid}`（不是前端記憶體或 localStorage），所以之後真的要交代「誰同意過」時，資料庫裡查得到。

## ⚠️ 上線前必須確認（原型階段刻意留下的風險，不是遺漏）

這些不是「之後有空再做」的優化項，是把真實客戶資料放進系統之前的最低門檻：

1. **時效自動算期的規則表尚未經律師覆核。** 見 `src/data/deadlineRules.js` 開頭的說明——目前是一般性研究整理的草稿，`在途期間`、`例假日順延`、`已聲請展延` 都沒有計算，UI 上也刻意標示為草稿並要求人工按「確認」。正式使用前務必請律師逐條核對規則與法條依據，並把 `RULES_STATUS` 改成 `LAWYER_REVIEWED`。

2. **Firestore / Storage 規則目前只檢查「有沒有登入」，沒有案件層級的權限區隔。** 見 `firestore.rules` 與 `storage.rules` 裡的 TODO——目前任何登入的員工都能讀寫所有客戶、案件、卷宗檔案。事務所通常需要利益衝突迴避 / 倫理牆，這必須在資料庫規則層做，不能只靠前端 UI 隱藏選單。`storage.rules` 尤其是全案風險最高的檔案——設錯規則等同洩漏受特權保護的卷宗。

3. **沒有稽核軌跡。** 誰在何時開過哪份卷宗、誰確認了哪筆時效，目前只記錄一個 uid，沒有做成可查詢的稽核紀錄或存取記錄。法律圈常因卷證保管鏈或內部調查需要這個。

4. **原型使用匿名登入（anonymous auth）。** 上線前需要換成真實的員工帳號機制（Email/密碼、SSO 或自訂 claims），並在此基礎上做角色權限。

5. **已修過的一個真實 bug，留作提醒：** 期限計算原本用 `toISOString().slice(0,10)`，在 UTC+8（台灣）時區會把日期算錯一天（凌晨時段尤其明顯）。已改用讀取本地年月日的 `formatLocalISO()`，並補了 `tests/deadlineRules.test.mjs` 防止回歸。修改這段邏輯前，先確認 `npm test` 還是全綠。

6. **AI 閱卷模組的分析內容是靜態示範，不是真的在讀你的卷宗。** 要讓它變成真的能用的功能，需要另外設計「PDF 內容怎麼切段餵給模型、模型輸出怎麼標註原文/推論/生成」的機制——這是一個 RAG/grounding 工程問題，比排版本身難得多，也是「引用溯源」這個功能本身能不能被信任的關鍵。逐行溯源（不只是跳頁）也還沒做，見上方「AI 閱卷與策略分析模組」說明。
